# Plan 03: Create Standalone Regression Flows

**Phase:** 21 -- Maestro UI Testing Infrastructure
**Goal:** Create three independently executable Maestro regression flows covering library navigation, playback, and download journeys
**Requirements:** TESTING-04, TESTING-05

## Context

This plan creates the three standalone regression flows per D-05 and D-06. Each flow is independently executable via `maestro test .maestro/<file>.yaml`. Each flow begins with `launchApp` and calls `_login.yaml` as its first subflow to ensure authentication. Flows use testIDs from Plan 21-01 and subflows from Plan 21-02.

Per D-07, the download flow covers download success verification only -- no airplane mode / offline playback testing. Per D-08, all files are flat in `.maestro/` with no subdirectories.

## Prerequisites

**Plan 21-01** (testID attributes) and **Plan 21-02** (subflows) must be complete.

## Tasks

### Task 1: Create `library-navigation.yaml` regression flow

**File(s):** `.maestro/library-navigation.yaml`

**Action:** Create a standalone flow that tests the library navigation journey: authenticate, navigate to Library, browse items, tap an item, assert the item detail screen appears.

**File content structure:**

```yaml
appId: cloud.madtown.sideshelf
---
# Regression flow: Library navigation
# Verifies: login -> Library tab -> browse items -> tap item -> item detail screen
- launchApp
- runFlow: .maestro/_login.yaml
- tapOn: "Library"
- waitForAnimationToEnd
# Double-tap to ensure list root
- tapOn: "Library"
- waitForAnimationToEnd
# Verify library items are visible
- assertVisible:
    id: "library-item"
# Tap the first library item
- tapOn:
    id: "library-item"
    index: 0
- waitForAnimationToEnd
# Assert item detail screen is showing -- the Play button or Resume button should be visible
- assertVisible:
    id: "play-resume-button"
    optional: true
```

Key points:

- Starts with `launchApp` (standalone flow, not a subflow)
- Calls `_login.yaml` for idempotent authentication
- Uses `id: "library-item"` (existing testID, per D-04) for item selection
- Asserts the item detail screen by checking for the play-resume-button testID
- Does NOT call `_start-playback.yaml` -- this flow tests navigation only, not playback

### Task 2: Create `playback.yaml` regression flow

**File(s):** `.maestro/playback.yaml`

**Action:** Create a standalone flow that tests the playback journey: authenticate, start playback via the `_start-playback` subflow, open the full-screen player, verify controls (pause, seek, speed), then dismiss.

Per D-06, this flow covers: open item, start playback, pause, seek, skip chapter, assert floating player progress.

Per Research Pitfall 3, the seek slider uses `PanResponder` -- use `swipe` on the `seek-slider` element, not `tapOn`.

**File content structure:**

```yaml
appId: cloud.madtown.sideshelf
---
# Regression flow: Playback
# Verifies: login -> start playback -> full-screen player -> pause -> seek -> speed control -> dismiss
- launchApp
- runFlow: .maestro/_login.yaml
- runFlow: .maestro/_start-playback.yaml

# Navigate to Home to access FloatingPlayer (it hides on item detail page)
- tapOn: "Home"
- waitForAnimationToEnd

# Assert floating player is visible
- assertVisible:
    id: "floating-player"

# Open full-screen player
- tapOn:
    id: "floating-player"
- waitForAnimationToEnd

# Verify seek slider is present
- assertVisible:
    id: "seek-slider"

# Test seek by swiping on the slider (tapOn won't work with PanResponder)
- swipe:
    id: "seek-slider"
    direction: RIGHT
    duration: 400

# Verify speed control is present
- assertVisible:
    id: "speed-control"

# Dismiss full-screen player
- tapOn:
    id: "player-done-button"
- waitForAnimationToEnd

# Assert we're back and floating player is still showing
- assertVisible:
    id: "floating-player"
```

Key points:

- Uses `_start-playback.yaml` subflow for item selection and play
- Navigates to Home before tapping floating-player (FloatingPlayer hides on item detail page -- same pattern as capture-screenshots.yaml)
- Uses `swipe` with `direction: RIGHT` on `seek-slider` instead of `tapOn` (Research Pitfall 3)
- Tests speed-control visibility (tapping it opens a native UIMenu which Maestro cannot inspect further -- assertVisible is sufficient)
- Dismisses via `player-done-button` testID

### Task 3: Create `download.yaml` regression flow

**File(s):** `.maestro/download.yaml`

**Action:** Create a standalone flow that tests the download journey: authenticate, navigate to an item, tap the download trigger (ellipsis menu), tap "Download" from the native menu, assert download progress appears.

Per D-07, this covers download success verification only -- no offline/airplane mode testing.

Per Research Pitfall 4, `MenuView` presents as a native iOS `UIMenu`. Maestro taps the trigger testID to open the menu, then taps the menu item by its text label.

**File content structure:**

```yaml
appId: cloud.madtown.sideshelf
---
# Regression flow: Download
# Verifies: login -> navigate to item -> trigger download -> assert download progress
- launchApp
- runFlow: .maestro/_login.yaml

# Navigate to Library and open first item
- tapOn: "Library"
- waitForAnimationToEnd
- tapOn: "Library"
- waitForAnimationToEnd
- tapOn:
    id: "library-item"
    index: 0
- waitForAnimationToEnd

# Tap the ellipsis menu trigger to open the action menu
- tapOn:
    id: "download-button"
- waitForAnimationToEnd

# Tap "Download" from the native UIMenu
# (Maestro can tap menu items by text after the menu opens)
- tapOn: "Download"
- waitForAnimationToEnd

# Assert some download indication is visible
# The download progress view or a badge should appear after tapping Download.
# Use a generous wait -- downloads take time to start.
- waitForAnimationToEnd
```

Key points:

- Does NOT use `_start-playback.yaml` -- the download journey navigates to an item but does not need to play it
- Taps `download-button` testID (the ellipsis menu trigger from Plan 21-01), then taps "Download" text from the native menu
- The assertion is lenient -- native menu interaction and download progress display timing varies. The flow succeeding without error (no element-not-found failures) is the primary verification.
- No airplane mode / offline verification (D-07)
