# Phase 17.1: Add Span Tracing Debugging Aid - Research

**Researched:** 2026-03-13
**Domain:** In-process ring-buffer tracing, React Native file I/O, UI instrumentation
**Confidence:** HIGH

## Summary

This phase adds a self-contained span-tracing library to the app and instruments two debug domains: player state restoration (`player.restore.*`) and state machine transitions (`player.machine.*`). The library is a complete, ready-to-copy TypeScript implementation documented in `docs/plans/span-tracing/reference implementation.md`. All major design decisions are locked in CONTEXT.md — the implementation is primarily transcription + integration work, not novel design.

The three primary bugs targeted are (1) intermittent state machine transition rejections and (2) position-resets-to-zero after cold start. Both are race-condition bugs that standard logging cannot sequence reliably. A ring-buffer trace with monotonic `sequence` counters and correlated `restoreSessionId` makes causality visible across async boundaries.

The phase has five natural work areas: (A) implement `trace.ts` + `traceHooks.ts`, (B) configure at startup, (C) instrument `ProgressRestoreCollaborator`, (D) instrument `PlayerStateCoordinator`, (E) wire dump-write on rejection + long-press + UI viewer.

**Primary recommendation:** Implement the reference `LocalTrace` class verbatim, then add instrumentation domain by domain (restore first, machine second), then wire UI. Each area can be a separate plan.

### Pattern 1: Singleton trace initialization

The `trace` singleton is created at module load time (zero config). `initializeApp()` calls `trace.configure(...)` to override defaults before any service code runs.

```typescript
// src/index.ts — near top of initializeApp(), before playerService.initialize()
import { trace } from "@/lib/trace";

trace.configure({
  bufferSize: 800,
  maxEventsPerSpan: 30,
  redactKeys: ["token", "authorization", "password", "cookie"],
});
```

**Important:** `trace.configure()` must run before `playerService.initialize()` and before the restore flow, so that all events during cold-start boot are captured.

### Pattern 2: Span tree for restore flow

The restore root span wraps the entire `restorePlayerServiceFromSession()` method. Child spans are started/ended explicitly (not `withSpan`) because the code has multiple early-return paths that need the span to still be ended.

```typescript
// src/services/player/ProgressRestoreCollaborator.ts
import { trace } from '@/lib/trace';

async restorePlayerServiceFromSession(): Promise<void> {
  const restoreSessionId = Math.random().toString(16).slice(2);
  const rootSpan = trace.startSpan('player.restore.session', { restoreSessionId });

  try {
    // ... source checks, each in a child span
    const dbSpan = trace.startSpan('player.restore.source.db', {}, rootSpan.context);
    // ... query
    trace.endSpan(dbSpan, 'ok', { found: true, itemId, positionMs });

    // ... reconcile span, apply span, verify span

    trace.endSpan(rootSpan, 'ok');
  } catch (error) {
    trace.recordError(error, rootSpan);
    trace.endSpan(rootSpan, 'error');
    throw error;
  }
}
```

### Pattern 3: addEvent for machine transitions

Machine trace calls go directly into `handleEvent()` (formerly `_processEvent()`). The coordinator already has the event, validation result, and context snapshot — all the data needed. No span is needed; standalone events with full attribute payloads are sufficient and cheaper.

```typescript
// Inside PlayerStateCoordinator.handleEvent(), after validateTransition()
import { trace } from '@/lib/trace';

// Always emit event.received
trace.addEvent('player.machine.event.received', {
  sequence: this.machineSequence++,
  source: event.source ?? 'unknown',
  event: event.type,
  fromState: currentState,
  itemId: this.context.currentTrack?.libraryItemId,
  positionMs: this.context.position,
  restoreSessionId: event.restoreSessionId,
});

if (validation.allowed) {
  trace.addEvent('player.machine.transition.accepted', { ... });
} else {
  trace.addEvent('player.machine.transition.rejected', {
    reason: validation.reason,
    ...
  });
  // Auto-dump on rejection
  this._writeDumpToDisk('rejection', event).catch(() => {});
}
```

### Pattern 4: Auto-dump file write

Uses `expo-file-system` `File` API directly, following the same pattern as `exportUtils.ts` but writing to Documents directory (persistent).

```typescript
import { File, Paths } from "expo-file-system";
import { trace } from "@/lib/trace";

async function writeDumpToDisk(
  reason: "rejection" | "manual",
  rejectionEvent?: unknown
): Promise<string> {
  const iso = new Date().toISOString().replace(/:/g, "-").replace(/\./g, "-");
  const filename = `trace-dump-${iso}.json`;
  const file = new File(Paths.document, filename);

  const payload = trace.exportTrace({
    appVersion: Application.nativeApplicationVersion ?? "unknown",
    platform: Platform.OS,
    dumpReason: reason,
    rejectionEvent: rejectionEvent ?? null,
  });

  await file.write(JSON.stringify(payload, null, 2));
  return file.uri;
}
```

Note: `expo-application` is already an Expo SDK package available without install.

### Pattern 6: TraceDumps viewer component

Follows CoordinatorDiagnostics visual style. Reads dump files from Documents directory on mount. Lists rows with timestamp + size + Share + Delete. "Clear All Dumps" at section top.

```typescript
// src/components/diagnostics/TraceDumps.tsx
// Lists: new File(Paths.document, ...) files matching 'trace-dump-*.json'
// Share: exportJsonAsFile() from exportUtils.ts
// Delete: file.delete()
```

**Key implementation note:** `expo-file-system` v2 (`File`, `Directory`, `Paths`) does not have a native directory-listing API in the same way as the legacy `FileSystem.readDirectoryAsync()`. The project already uses the newer API (`import { File, Paths } from 'expo-file-system'`). Directory listing requires `new Directory(Paths.document).list()` — verify this API exists on the installed version.

### Anti-Patterns to Avoid

- **Wrapping machine events in spans:** Each machine event is a point-in-time record, not a duration. Use `trace.addEvent()` not `trace.withSpan()` for machine events.
- **Redacting `itemId` / `positionMs`:** These are not secrets. Only redact the keys in `redactKeys` config (`token`, `authorization`, `password`, `cookie`).
- **Awaiting dump writes on the hot path:** Auto-dump on rejection must be fire-and-forget (`.catch(() => {})`) — the coordinator's event processing lock must not be held during a file write.
- **Importing `trace` from a barrel that re-exports services:** `trace` is a leaf module with no dependencies. Import it directly: `import { trace } from '@/lib/trace'`. Never import through a barrel that includes service singletons.
- **Object-returning selectors in TraceDumps viewer:** Use individual `useAppStore` selectors, not `{ a, b } = useAppStore(s => ({ a, b }))`.

## Don't Hand-Roll

| Problem                | Don't Build                              | Use Instead                                                           | Why                                                  |
| ---------------------- | ---------------------------------------- | --------------------------------------------------------------------- | ---------------------------------------------------- |
| Ring buffer overflow   | manual array shift                       | `RingBuffer` class in `trace.ts`                                      | already handles wrap-around correctly                |
| Attribute sanitization | custom JSON replacer                     | `sanitizeValue()` in `trace.ts`                                       | handles depth, strings, bigint, functions, redaction |
| Error serialization    | `error.toString()`                       | `serializeError()` in `trace.ts`                                      | preserves stack, cause, name                         |
| Share sheet            | custom Share UI                          | `exportJsonAsFile()` from `exportUtils.ts`                            | already handles iOS/Android sharing                  |
| File write             | `FileSystem.writeAsStringAsync` (legacy) | `new File(Paths.document, name).write(content)`                       | matches project's v2 API pattern                     |
| Haptic feedback        | setTimeout shimmer                       | `Haptics.impactAsync(ImpactFeedbackStyle.Medium)` from `expo-haptics` | one-liner, platform-correct                          |

## Common Pitfalls

### Pitfall 1: Context stack contamination across async boundaries

**What goes wrong:** `trace.withSpan()` pushes a context onto a stack. If an `await` inside the span callback is interrupted by another microtask that also calls `trace.startSpan()`, the inner span may inherit the wrong parent.

**Why it happens:** The `contextStack` is module-level state, not per-async-context (no AsyncLocalStorage in React Native hermes).

**How to avoid:** Pass `parentContext` explicitly to `trace.startSpan(name, attrs, parentContext)` for all child spans in the restore flow — do not rely on implicit context propagation across `await` boundaries.

**Warning signs:** Spans appearing with wrong parent span IDs in exported JSON.

### Pitfall 2: `restoreSessionId` threading

**What goes wrong:** The restore session ID is generated once at the top of `restorePlayerServiceFromSession()`. If it is stored in a module-level variable, a concurrent restore attempt (edge case) overwrites the ID, making correlation unreliable.

**How to avoid:** Thread `restoreSessionId` as a local variable and pass it explicitly to each child span and to `dispatchPlayerEvent()` calls that need it. If using a module-level variable, document the assumption that restore is single-flight.

### Pitfall 3: `source` field on PlayerEvent

**What goes wrong:** `PlayerEvent` union type in `src/types/coordinator.ts` does not currently have a `source` field. Adding `source` to individual event variants (e.g., `| { type: "PLAY"; source?: EventSource }`) requires touching every event variant or the union discriminant.

**How to avoid:** The simplest approach (Claude's Discretion) is to add `source?: EventSource` as an optional field on every event variant, or to use a wrapper type `{ event: PlayerEvent; source: EventSource }` only at the coordinator's `dispatch()` boundary. The coordinator's `handleEvent()` receives the source from the wrapper, not from the individual event type.

**Recommended approach:** Add `source?: EventSource` and `restoreSessionId?: string` to a base intersection applied to all events — OR use a separate `DispatchMeta` parameter on `dispatch(event, meta?)`. The latter avoids touching all event variant definitions.

### Pitfall 4: Directory listing API (expo-file-system v2)

**What goes wrong:** The project uses the v2 `File`/`Directory` API (`import { File, Paths } from 'expo-file-system'`). Listing files in a directory uses `new Directory(uri).list()`, which returns `(File | Directory)[]`. The legacy `FileSystem.readDirectoryAsync()` should not be mixed in.

**How to avoid:** Use `const dir = new Directory(Paths.document); const items = dir.list();` and filter by `file instanceof File && file.name.startsWith('trace-dump-')`.

**Warning signs:** TypeScript errors importing from `expo-file-system/build/...` paths, or runtime errors if mixing legacy and v2 APIs.

### Pitfall 5: Auto-dump blocking coordinator lock

**What goes wrong:** `handleEvent()` runs inside `lock.acquire("state-transition", ...)`. If `_writeDumpToDisk()` is awaited inside the lock, the coordinator stalls for the duration of the file write (potentially hundreds of ms), causing event queue backup.

**How to avoid:** Fire-and-forget: `this._writeDumpToDisk('rejection', event).catch((err) => log.warn(...));`. Do not await.

### Pitfall 6: `onRecord` hook for real-time rejection detection

**What goes wrong:** The `LocalTrace.configure()` accepts an `onRecord` callback that fires synchronously on every `pushRecord`. If auto-dump is triggered from `onRecord`, it runs on the same callstack as `trace.addEvent()`, which may be inside the coordinator lock.

**How to avoid:** Use the `onRecord` hook to set a flag or schedule via `setImmediate()` / `Promise.resolve().then()`, not to directly await file I/O. Alternatively, trigger the dump from `handleEvent()` after the trace call — which is where the rejection is already detected — rather than from `onRecord`.
