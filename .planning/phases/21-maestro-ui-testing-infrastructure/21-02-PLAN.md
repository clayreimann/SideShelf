# Plan 02: Create Reusable Maestro Subflows (\_login and \_start-playback)

**Phase:** 21 -- Maestro UI Testing Infrastructure
**Goal:** Create the `_login.yaml` and `_start-playback.yaml` reusable subflows that standalone regression flows will consume
**Requirements:** TESTING-02, TESTING-04

## Context

This plan creates the two reusable Maestro subflows in `.maestro/`. The `_login.yaml` subflow authenticates using environment variable credentials and is idempotent (D-10) -- it skips login if the user is already on the home/library screen. The `_start-playback.yaml` subflow opens the first library item and starts playback. Both follow the existing `capture-screenshots.yaml` patterns (appId header, `waitForAnimationToEnd`, `id:` selectors).

Per D-08, files are placed flat in `.maestro/` with underscore prefix convention. Per D-02, credentials use `${MAESTRO_USERNAME}`, `${MAESTRO_PASSWORD}`, `${MAESTRO_SERVER_URL}` env var syntax.

## Prerequisites

**Plan 21-01** must be complete -- the subflows target testIDs added in that plan (`login-server-url-input`, `login-username-input`, `login-password-input`, `login-button`, `play-resume-button`).

## Tasks

### Task 1: Create `_login.yaml` idempotent login subflow

**File(s):** `.maestro/_login.yaml`

**Action:** Create the idempotent login subflow per D-10 and Research Pattern 3.

The flow detects "already logged in" by checking if "Library" text (tab bar label) is visible. If not visible, it fills credentials and submits. Per Research Pitfall 2, the login form's submit button is disabled until the server URL field blurs and the async ping completes -- so the sequence must tap the username field after entering the server URL (triggering blur/ping), then wait before tapping login.

**File content structure:**

```yaml
appId: cloud.madtown.sideshelf
---
# Idempotent login subflow
# If already on home/library screen, skip login.
# Credentials injected via --env flags from .env.maestro.
- runFlow:
    when:
      notVisible: "Library"
    commands:
      - tapOn:
          id: "login-server-url-input"
      - clearText
      - inputText: ${MAESTRO_SERVER_URL}
      - tapOn:
          id: "login-username-input"
      - waitForAnimationToEnd
      - clearText
      - inputText: ${MAESTRO_USERNAME}
      - tapOn:
          id: "login-password-input"
      - clearText
      - inputText: ${MAESTRO_PASSWORD}
      - waitForAnimationToEnd
      - tapOn:
          id: "login-button"
      - waitForAnimationToEnd
```

Key points:

- `appId` header is required even for subflows (Research Pitfall 1)
- No `launchApp` -- subflows do not launch the app (the standalone parent flow does)
- The tap on `login-username-input` after entering the server URL triggers `onBlur` on the URL field, which fires `tryPing()`. The `waitForAnimationToEnd` after that gives the ping time to resolve before entering credentials.
- The final `waitForAnimationToEnd` after tapping login waits for navigation to the home screen.

### Task 2: Create `_start-playback.yaml` subflow

**File(s):** `.maestro/_start-playback.yaml`

**Action:** Create a subflow that navigates to the Library tab, taps the first library item, and starts playback. Follow the existing `capture-screenshots.yaml` pattern for item selection (index-based).

**File content structure:**

```yaml
appId: cloud.madtown.sideshelf
---
# Subflow: open first library item and start playback
- tapOn: "Library"
- waitForAnimationToEnd
# Double-tap Library tab to ensure we're at the list root
# (same pattern as capture-screenshots.yaml)
- tapOn: "Library"
- waitForAnimationToEnd
- tapOn:
    id: "library-item"
    index: 0
- waitForAnimationToEnd
# Tap Play or Resume (label varies depending on prior state)
- tapOn:
    id: "play-resume-button"
    optional: true
- tapOn:
    text: "Resume"
    optional: true
- waitForAnimationToEnd
```

Key points:

- No `launchApp` -- this is a subflow called after login
- Double-tap Library tab resets any deep navigation stack (matches existing capture-screenshots.yaml pattern)
- Uses `id: "play-resume-button"` (testID from Plan 21-01) with `optional: true` because the button is only visible when not currently playing
- Falls back to tapping "Resume" text with `optional: true` for the case where playback was previously started
- `waitForAnimationToEnd` after each navigation and the play tap
