# Milestone v1.2: Tech Cleanup

**Status:** ✅ SHIPPED 2026-03-08
**Phases:** 10–13
**Total Plans:** 8

## Overview

Internal codebase quality milestone. No user-facing features. Four targeted cleanup phases addressed technical debt accumulated across v1.0 and v1.1: SQLite performance, redundant state fetches, service file sizes, and a private fork dependency.

## Phases

### Phase 10: DB Quick Wins

**Goal**: Database performance is measurably improved through WAL mode, complete index coverage on all foreign-key query paths, and elimination of N+1 upsert loops
**Depends on**: Nothing (first v1.2 phase)
**Requirements**: DB-01, DB-02, DB-03, DB-04, DB-05, DB-06, DB-07
**Plans**: 2 plans

Plans:

- [x] 10-01-PLAN.md — WAL pragma, blocking DbErrorScreen, and 4 missing indexes via schema + migration (DB-01–05)
- [x] 10-02-PLAN.md — N+1 batch upsert conversion across all db/helpers/ files + test assertions (DB-06–07)

**Details:**

- WAL journal mode + `synchronous=NORMAL` set via `execSync` on raw SQLite handle before Drizzle wraps connection
- 4 FK indexes added: `library_items.library_id`, `media_metadata.library_item_id`, `audio_files.media_id`, `media_progress(user_id, library_item_id)`
- `DbErrorScreen` blocking error component — uses basic RN primitives (safe when DB is broken); disk-full errors suppress reset button
- Batch upserts: `upsertLibraryItems`, `upsertLibraries`, `upsertGenres`, `upsertNarrators`, `upsertTags` all converted to single `INSERT ON CONFLICT DO UPDATE` calls
- `upsertLibraryItemTx` removed — zero callers outside its own file

### Phase 11: useEffect Cleanup + State Centralization

**Goal**: Redundant mount-time DB fetches and AsyncStorage reads are eliminated; shared preference and entity data lives in Zustand slices and survives navigation
**Depends on**: Phase 10
**Requirements**: EFFECT-01–06, STATE-01–03
**Plans**: 2 plans

Plans:

- [x] 11-01-PLAN.md — Slice extensions: viewMode in settingsSlice, progressMap in seriesSlice, getOrFetchAuthorById in authorsSlice, availableTags in loggerSlice (STATE-01–03, EFFECT-06)
- [x] 11-02-PLAN.md — Consumer wiring: userId in useAuth, authorId/seriesId in CachedItemDetails, remove redundant useEffects, logout DB wipe (EFFECT-01–05)

**Details:**

- `viewMode` moved to `settingsSlice` with AsyncStorage persistence (consistent with all other settings)
- `progressMap` in `seriesSlice` — SeriesDetailScreen loads all book progress in one bulk query (no N+1 per-book mount effects)
- `getOrFetchAuthorById` in `authorsSlice` — in-memory find() first, DB fallback; eliminates component-level DB calls
- `availableTags` in `loggerSlice` — `getAllTags()` called once; shared by LogsScreen and LoggerSettingsScreen
- `userId` added to `useAuth()` — 5+ components no longer call `getUserByUsername()` DB reads
- `authorId`/`seriesId` resolved inside `libraryItemDetailsSlice.fetchItemDetails` — no separate per-item useEffect
- `MoreScreen` app version reads replaced with synchronous module-scope constant
- `wipeUserData` deletes content tables in child-before-parent FK order; preserves users and logger tables

### Phase 12: Service Decomposition

**Goal**: PlayerService and DownloadService are split into a public facade plus focused private collaborators — the coordinator dispatch contract, singleton interface, and all existing tests remain intact
**Depends on**: Phase 11
**Requirements**: DECOMP-01, DECOMP-02
**Plans**: 2 plans

Plans:

- [x] 12-01-PLAN.md — PlayerService facade + 4 collaborators (TrackLoading, PlaybackControl, ProgressRestore, BackgroundReconnect) + 92% coverage (DECOMP-01)
- [x] 12-02-PLAN.md — DownloadService facade + 2 collaborators (DownloadStatus, DownloadRepair) + 91% coverage (DECOMP-02)

**Details:**

- `IPlayerServiceFacade` interface in `types.ts` — both PlayerService and collaborators import from one file; no collaborator imports PlayerService directly
- `rebuildCurrentTrackIfNeeded` placed in `ProgressRestoreCollaborator`; exposed on facade so `PlaybackControlCollaborator` can call it without a circular import
- `BackgroundReconnectCollaborator` keeps `require()` pattern for `PlayerBackgroundService` (module cache clearing in DEV mode)
- `DownloadService` collaborators are stateless — simpler than PlayerService pattern; they query DB/filesystem independently with no callbacks to facade
- `isDownloadActive` and `getDownloadStatus` remain as direct Map reads in facade — delegating would break `activeDownloads` isolation goal
- `forceExit: true` added to `jest.config.js` — `@react-native-community/netinfo` starts a reachability timer that never unrefs; fix applied without changing test behavior

### Phase 13: RN Downloader Migration

**Goal**: The custom `spike-event-queue` fork is replaced by mainline `@kesha-antonov/react-native-background-downloader@4.5.3`; downloads continue to work correctly including restart recovery and iCloud exclusion
**Depends on**: Phase 12
**Requirements**: DWNLD-01, DWNLD-02, DWNLD-03, DWNLD-04
**Plans**: 2 plans

Plans:

- [x] 13-01-PLAN.md — Fork diff investigation document: API surface diff, call site mapping, event queue analysis, iCloud exclusion independence, smoke test checklist (DWNLD-01)
- [x] 13-02-PLAN.md — Package swap + DownloadService API migration + startup reconciliation unit tests (DWNLD-02–04)

**Details:**

- Fork diff investigation: `docs/investigation/rnbd-fork-diff.md` — full API surface diff, call site mapping, smoke test checklist
- No migration flag needed — existing repair/reconciliation flow handles fork-era in-progress downloads (beta app, aggressive cleanup acceptable)
- `void` prefix on pause/resume/stop — fire-and-forget is functionally safe
- `checkForExistingDownloads` → `getExistingDownloadTasks` (mainline rename); all other method names preserved
- Handlers-before-`.start()` is a critical invariant — documented with `CRITICAL` comment in DownloadService
- `^4.5.3` semver range (npm default); exact version enforced by `package-lock.json`
- iCloud exclusion plugin (`withExcludeFromBackup`) operates independently of RNBD — v1.1 behavior not regressed
- iCloud exclusion decode bug (paths with `%20`) discovered during smoke test — fixed in 13-02

## Milestone Summary

**Key Decisions:**

- WAL pragma via `execSync` (raw SQLite handle) — must run before Drizzle wraps connection; `synchronous=NORMAL` is connection-level so set in `getSQLiteDb()` guard
- `IPlayerServiceFacade` in `types.ts` — prevents circular imports between facade and collaborators
- DownloadService collaborators are stateless (no facade reference) — simpler than PlayerService pattern
- No migration flag for fork → mainline transition — repair/reconciliation handles it; beta app so aggressive cleanup is acceptable
- `forceExit: true` in jest.config.js — fixes lint-staged `--bail` without changing test behavior

**Issues Resolved:**

- 500-item library sync was executing 1,000+ sequential DB queries — resolved by batch upsert conversion
- PlayerService (~1,700 lines) and DownloadService (~900 lines) were monolithic — decomposed to facades + collaborators
- App was dependent on a private RNBD fork (`spike-event-queue`) — replaced with mainline 4.5.3
- iCloud exclusion decode bug (paths with `%20`) discovered and fixed during smoke test

**Issues Deferred:**

- Path standardization (encoding mismatch: `file://` URIs, POSIX paths, `D:/C:` DB prefixes) — captured as pending todo
- Orphan download reassociation UI — deferred
- `PERF-01`: `NATIVE_PROGRESS_UPDATED` bypass of async-lock — needs safety analysis
- Expo SDK 55 upgrade — gated on RNTP Android bridgeless compatibility (issue #2443)
- `ProgressService` decomposition (DECOMP-03) — deferred; background service contract complexity warrants standalone phase

**Technical Debt Incurred:**

- None significant — this milestone was specifically designed to reduce debt
