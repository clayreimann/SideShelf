# Plan 04: Project Setup -- Env File, Gitignore, and npm Script

**Phase:** 21 -- Maestro UI Testing Infrastructure
**Goal:** Set up the credential management infrastructure (`.env.maestro` + `.env.maestro.example`), update `.gitignore` to exclude credentials, and add the `maestro:test` npm script
**Requirements:** TESTING-02, TESTING-05

## Context

This plan creates the project-level configuration that enables running the Maestro regression suite. Per D-01, credentials are stored in `.env.maestro` (gitignored) with `.env.maestro.example` committed as a template. Per D-02, env vars are `MAESTRO_USERNAME`, `MAESTRO_PASSWORD`, `MAESTRO_SERVER_URL`. Per D-09, an npm script `maestro:test` runs all three standalone flows in sequence.

Per Research Pitfall 5, `.env.maestro` is NOT covered by the existing `.env*.local` gitignore pattern and must be added explicitly.

This plan can run in parallel with Plans 21-01 through 21-03 (Wave 1) since it only touches config files, not source or YAML.

## Prerequisites

None -- this plan touches only config files (`.gitignore`, `package.json`, `.env.maestro.example`) and creates a shell script. It is independent of the testID and YAML plans, though the `maestro:test` script will only work after Plans 21-01 through 21-03 are complete.

## Tasks

### Task 1: Create `.env.maestro.example` and update `.gitignore`

**File(s):**

- `.env.maestro.example` (new file)
- `.gitignore` (modify)

**Action:**

1. Create `.env.maestro.example` with placeholder values and explanatory comments:

```bash
# Maestro UI test credentials
# Copy this file to .env.maestro and fill in real values.
# .env.maestro is gitignored -- never commit real credentials.
MAESTRO_SERVER_URL=http://your-abs-server.local:13378
MAESTRO_USERNAME=your-username
MAESTRO_PASSWORD=your-password
```

2. Add `.env.maestro` to `.gitignore`. Place it immediately after the existing `# local env files` / `.env*.local` block to keep env-related ignores together:

```
# local env files
.env*.local
.env.maestro
```

3. Verify `.env.maestro.example` is NOT gitignored (it should be committed):
   - Run `git check-ignore .env.maestro.example` -- should return nothing (not ignored)
   - Run `git check-ignore .env.maestro` -- should return `.env.maestro` (ignored)

### Task 2: Create `scripts/run-maestro-tests.sh` and add `maestro:test` npm script

**File(s):**

- `scripts/run-maestro-tests.sh` (new file)
- `package.json` (modify)

**Action:**

1. Create `scripts/run-maestro-tests.sh` -- a shell script that sources `.env.maestro` and runs all three standalone flows in sequence. Follow the pattern of the existing `scripts/run-screenshots.sh` (uses `set -euo pipefail`, `REPO_ROOT` variable).

```bash
#!/usr/bin/env bash
# Runs all Maestro regression flows in sequence.
# Credentials are sourced from .env.maestro (see .env.maestro.example).
set -euo pipefail

REPO_ROOT="$(cd "$(dirname "${BASH_SOURCE[0]}")/.." && pwd)"
ENV_FILE="${REPO_ROOT}/.env.maestro"

if [[ ! -f "${ENV_FILE}" ]]; then
  echo "ERROR: ${ENV_FILE} not found."
  echo "Copy .env.maestro.example to .env.maestro and fill in your credentials."
  exit 1
fi

# Source credentials
set -a
# shellcheck source=/dev/null
. "${ENV_FILE}"
set +a

ENV_FLAGS=(
  --env "MAESTRO_USERNAME=${MAESTRO_USERNAME}"
  --env "MAESTRO_PASSWORD=${MAESTRO_PASSWORD}"
  --env "MAESTRO_SERVER_URL=${MAESTRO_SERVER_URL}"
)

FLOWS=(
  library-navigation
  playback
  download
)

for flow in "${FLOWS[@]}"; do
  echo ">> Running ${flow}.yaml..."
  maestro test "${ENV_FLAGS[@]}" "${REPO_ROOT}/.maestro/${flow}.yaml"
  echo ">> ${flow}.yaml passed."
done

echo "All Maestro regression flows passed."
```

2. Make the script executable: `chmod +x scripts/run-maestro-tests.sh`

3. Add the `maestro:test` npm script to `package.json`. Place it near the existing `maestro:screenshots` script (if it exists) or after the test-related scripts:

```json
"maestro:test": "bash scripts/run-maestro-tests.sh"
```

This approach (shell script called from npm) is cleaner than inlining the full command in package.json, follows the existing `run-screenshots.sh` pattern, and handles the `.env.maestro` sourcing plus error handling.

4. Verify the script exists and is executable: `ls -la scripts/run-maestro-tests.sh`
