# Phase 17.1: Add span tracing debugging aid - Context

**Gathered:** 2026-03-13
**Status:** Ready for planning

<domain>
## Phase Boundary

Add a span tracing library (`trace.ts` + React hooks) to the codebase, configure it at startup, instrument two debug domains (`player.restore.*` and `player.machine.*`), and ship a trace dump viewer UI in More > Track Player. This is a debugging aid — not user-facing functionality.

Primary target bugs:

1. Rejected/unexpected state machine transitions (intermittent)
2. Progress resets to 0:00 after hitting Play post-cold-start (position appears restored, then jumps to beginning)

</domain>

<decisions>
## Implementation Decisions

### Library

- Implement `trace.ts` and `hooks.ts` exactly from `docs/plans/span-tracing/reference implementation.md`
- Place at `src/lib/trace.ts` and `src/lib/traceHooks.ts` (or `src/lib/trace/` if splitting makes sense)
- Configure at startup in `initializeApp()`: `bufferSize: 800`, `maxEventsPerSpan: 30`, `redactKeys: ['token', 'authorization', 'password', 'cookie']`
- Export singleton `trace` instance for use throughout the app

### player.restore.\* instrumentation

- **Full span tree** — not just events:
  - Root span: `player.restore.session` (wraps entire `restorePlayerServiceFromSession()`)
  - Child spans per source: `player.restore.source.db` (active sessions query), `player.restore.source.memory` (store check), `player.restore.source.track_player` (TrackPlayer queue check)
  - `player.restore.reconcile` span for the candidate selection logic
  - `player.restore.apply` span for the track/queue rebuild
  - `player.restore.verify` span for the final state check
- Each source span records: `found`, `itemId`, `positionMs`, `updatedAt`, `sessionId`
- Reconciliation span records decision events: `restore.candidate.accepted`, `restore.candidate.rejected`, `restore.decision.finalized` with `source`, `reason`, `positionMs`
- Thread `restoreSessionId` (generated per restore attempt) through all child spans and any machine events triggered during restore

### player.machine.\* instrumentation

- **All transitions traced** — accepted, rejected, and ignored
- Direct `trace.addEvent()` calls inside `PlayerStateCoordinator._processEvent()` (and side-effect dispatch methods)
- Every machine trace event carries:
  - `source` — who dispatched the event: `'ui' | 'restore' | 'native_player' | 'audio_focus' | 'remote_command' | 'sleep_timer' | 'startup_bootstrap'`
  - `sequence` — monotonic counter for absolute ordering within the coordinator session
  - `fromState`, `toState`, `reason` (on rejection)
  - `restoreSessionId` (when event originates from restore flow)
  - `itemId`, `chapterId`, `positionMs` from coordinator context snapshot
- Event names: `player.machine.event.received`, `player.machine.transition.accepted`, `player.machine.transition.rejected`, `player.machine.transition.ignored`, `player.machine.state.entered`
- Code paths that dispatch events (services, UI, native callbacks) pass `source` as part of dispatch metadata

### Auto-dump on rejection

- When a `player.machine.transition.rejected` event is recorded, immediately write the current ring buffer to a timestamped JSON file
- File naming: `trace-dump-{ISO8601}.json` (e.g., `trace-dump-2026-03-13T10-23-45-123Z.json`)
- Write to app documents directory via `expo-file-system` (same pattern as other file operations in `fileSystem.ts`)
- Append new file per dump — files accumulate until user clears them from the UI
- No maximum cap this phase — keep it simple

### Manual dump (long-press play/pause)

- Long-press on the play/pause button triggers a trace dump to disk (same timestamped file behavior)
- Available on **all build types** (not just `__DEV__`) — needed for TestFlight/production debugging
- Applies to all three surfaces where play/pause appears: FloatingPlayer, FullScreenPlayer, and ItemDetails player controls
- Brief haptic feedback or toast confirmation after dump written

### Trace dump viewer UI

- New "Trace Dumps" section added to More > Track Player screen, below existing CoordinatorDiagnostics content
- Lists all dump files in the documents directory, sorted newest first
- Each row shows: ISO timestamp, file size, Share button (native Share sheet via `expo-sharing`), delete button (swipe or icon)
- "Clear All Dumps" button at top of section
- Reuses `exportJsonAsFile` / `exportUtils.ts` patterns for Share sheet

### Buffer configuration

- 800 in-memory records (covers cold start restore + state machine events through first play tap)
- No AsyncStorage persistence — in-memory is sufficient for the target use case (cold start → first play → user notices reset → long-press dump)
- Buffer cleared only on explicit "Clear History" (existing coordinator flow) or app restart

### Claude's Discretion

- Exact `source` tagging mechanism for event dispatch (property on event payload vs separate argument to `dispatchPlayerEvent`)
- Whether `restoreSessionId` is stored in a module-level variable or threaded explicitly through function arguments
- File system path choice for dump files (DocumentDirectory vs CacheDirectory)
- Toast vs Alert vs silent for long-press confirmation
- Whether to split `trace.ts` into a subdirectory or keep as flat files in `src/lib/`

</decisions>

<specifics>
## Specific Ideas

- The position-resets-to-zero bug: position appears restored at cold start, but after pressing Play sometimes jumps to beginning. This is likely a race between restore-originated machine events and native player state. The `restoreSessionId` correlation is specifically intended to make this visible in the trace.
- Long-press play/pause for manual dump: reachable from any playback surface, available in all builds — so TestFlight installs can capture traces too.
- The `first use.md` and `reference implementation.md` in `docs/plans/span-tracing/` are authoritative references for the library implementation and instrumentation patterns. Planner/researcher should read both.
- Dump files should include a `meta` field with `appVersion`, `platform`, `dumpReason` ('rejection' | 'manual') and `rejectionEvent` details.

</specifics>
